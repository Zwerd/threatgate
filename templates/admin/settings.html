DXL / TIE Integration

When enabled, every Hash (MD5/SHA1/SHA256) submitted in ZIoCHub is sent to TIE on the DXL fabric as Known Malicious.

You need both of the following on the Linux server:

  1. Python packages dxlclient and dxltieclient - the code that talks to the DXL fabric and TIE. These are already in requirements.txt and are installed automatically when you run setup.sh (online or offline).
  2. ePO provisioning package - the config file (dxlclient.config) and certificate files (e.g. brokercerts.crt, client.crt, client.key) generated by ePO for this client. Without these, the client cannot authenticate to the broker. You must create this in ePO and place it on the server yourself.
Info: Installation and setup on Linux server

If you installed ZIoCHub with setup.sh (online or offline), the DXL Python packages are already installed. You only need to add the ePO package and set the config path below.

1. Python packages (only if you did not use setup.sh):
pip install -r requirements.txt or pip install dxlclient dxltieclient

Offline: On a machine with internet: pip download dxlclient dxltieclient -d ./wheels. Transfer the wheels folder to the server. On the server: pip install --no-index --find-links ./wheels dxlclient dxltieclient

2. ePO package: In ePO, create a DXL client provisioning package for Linux (DXL - Provisioning / Client Management - Linux). You will get a config file and certificate files. Either upload each file below (recommended), or copy them to the server and enter the config path manually.

Recommended path (manual): e.g. /opt/ziochub/certs/dxl/ - then the config path is /opt/ziochub/certs/dxl/dxlclient.config. Ensure the user running ZIoCHub can read all files in that directory.

Upload ePO files (one file per field)

Upload each file you received from ePO separately. No need to open or extract any archive. Files are saved to a fixed directory on this server and the config path is updated automatically. You can upload all four at once or one by one; upload only the files you have (e.g. if ePO gave you one bundle with a different name, use the field that matches its purpose).

The main config file (often named dxlclient.config or dxlclient.cfg). It contains broker addresses and paths to the certificate files. After upload, the paths inside it are rewritten to point to the other files you upload here, so they work on this server.

The broker CA certificate bundle (often brokercerts.crt or similar). Used to authenticate the DXL broker(s). Saved as brokercerts.crt in the fixed directory.

The client certificate for this DXL client (often client.crt). It identifies ZIoCHub to the broker. Saved as client.crt.

The private key that matches the client certificate (often client.key). Keep this secret. Saved as client.key with restricted permissions.

After upload, the "Path to dxlclient.config" field below is set automatically. You can then enable DXL and run Test Connection.

If you used the upload form above, this field is set automatically. Otherwise, enter the full path to the dxlclient.config file on this server (e.g. /opt/ziochub/certs/dxl/dxlclient.config or .../data/dxl/dxlclient.config). The process running ZIoCHub must have read access to that file and to the cert files referenced inside it.

Automation - Push YARA to external APIs

When an admin approves a YARA rule, the file is sent in the background to each configured target (any product that accepts YARA via HTTP and API key). Failures are written to logs; the user sees on-screen feedback (uploading, then SUCCESS or ERROR).

Add any product that accepts YARA via HTTP (e.g. FireEye NX, ESM, or other). For each: display name, base URL (e.g. https://product.local:8443), path (e.g. /api/v1/yara), and API key if required. Request: POST with body = YARA content, optional X-API-Key header.

Syslog / CEF Forwarding

Forward audit logs in CEF format via UDP to a SIEM or syslog server. Logs are always stored locally (48h rotation).
